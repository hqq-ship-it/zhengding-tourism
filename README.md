2\. 实践流程

本次实践以“正定文旅”前端原生 JS 项目为载体，完整体验了代码的版本控制与云端托管。

1\. 环境配置: 在本地 Windows 环境下安装 Git Bash，并通过 `git config --global` 命令完成了开发者姓名与邮箱的绑定。

2\. 本地仓库初始化: 使用命令 `mkdir zhengding-tourism` 创建项目目录，进入目录后执行 `git init` 成功建立本地 Git 仓库。

3\. 远程仓库协作: 在 GitHub 创建了同名的公开仓库，通过 `git remote add origin` 建立了本地与云端的连接，并将默认分支重命名为 `main`。

4\. 版本提交与推送: 编写了基础的前端代码，使用 `git add .` 将文件存入暂存区，遵循规范完成 3 次有效 `commit`，最后通过 `git push -u origin main` 成功推送到云端。

4\. 遇到的问题及解决方法



在本次实践中，我遇到了几个非常典型的命令行操作问题：



问题一：配置用户名时报错 `error: invalid key: user.nameXXX`

原因: 在输入 `git config --global user.name"我的名字"` 时，`user.name` 和双引号之间漏打了一个空格，导致 Git 无法识别该配置键名。

解决方法: 重新输入命令，严格保证格式，补充空格后（`user.name "我的名字"`）成功配置。



问题二：粘贴命令时终端报错“找不到命令”（如 `\\302\\226\\302\\226mkdir: command not found`）

原因: 从网页或其他富文本环境中复制命令时，连带复制了肉眼不可见的隐形格式控制符。Git Bash 终端识别这些隐藏乱码，导致正常命令失效。

解决方法: 放弃直接的 `Ctrl+V`，改用纯键盘手动输入短命令；对于长链接，先复制到系统“记事本”中过滤掉隐形格式，再通过鼠标右键的 `Paste` 或 `Shift + Insert` 安全粘贴到 Git Bash 中。



5\. Git 学习心得

通过本次完整的 Git 实践流程，我初步建立起了工程化开发的意识。以前在开发前端 UI 界面或是做原生 JS 项目时，每次遇到需求修改或是试验新功能，只能靠不断复制整个文件夹并重命名（比如最终版、绝对不改版）来备份。这种方式不仅占用空间，而且根本记不清每次具体改了哪几行代码。

掌握 Git 后，每一次 `commit` 都像是在时间线上存了一个精准的“档”，配合详尽的提交说明，让代码的历史演进一目了然。这对于全栈开发或是未来接入更复杂的媒体交互逻辑来说，是不可或缺的底层能力保障。

补充说明：进阶探索与好书推荐

1. Git 高级功能探索实践
在完成基础的提交与同步后，我进一步在本地项目探索了 Git 的进阶功能，模拟了实际工程开发中的常见工作流：
分支管理与合并 (Branch & Merge): 使用 `git checkout -b feature/books` 创建并切换到功能分支，在该分支上完成了读书笔记的编写。随后切换回 `main` 分支，通过 `git merge feature/books` 实现了代码的无缝合并。这种隔离式的开发能有效保障主线代码的稳定性。
版本回退 (Reset/Revert): 学习了使用 `git log` 查看提交历史与 Commit ID，并掌握了通过 `git reset --hard <commit-id>` 撤销错误修改、回滚到指定版本的方法。这为代码试错和问题定位提供了强大的安全网。
协作开发: 了解了通过拉取请求（Pull Request）进行代码审查，以及解决多人同时修改同一文件时产生的合并冲突（Conflict）的基础概念。
